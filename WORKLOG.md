## 2026-07-17 — ตัดสินกติกา: คะแนนสมรรถนะรวม = 0 แปลผลเป็น "เริ่มต้น"

> ⚠ **ยกเลิกแล้ว — อย่าลงมือทำตามแผนในหัวข้อนี้** ผู้ใช้ทบทวนใหม่ภายหลังแล้วตัดสินว่าไม่แก้
> (ดูบันทึก "ยกเลิกการแก้เกณฑ์ 0" ท้ายไฟล์ วันเดียวกัน)

- **ผู้ใช้ยืนยันแล้ว**: คะแนนสมรรถนะรวม `0` ให้แปลผลเป็น **"เริ่มต้น"** โดยขยายช่วงล่างของเกณฑ์
  กลางจาก `1–59` เป็น `0–59` (ปิดข้อค้างข้อ 4 ของรอบทดสอบ 2026-07-17 ที่ระบุว่าเอกสารจริงของ
  โรงเรียนไม่ได้ระบุการแปลผลคะแนน 0)
- เหตุผลที่ใช้ตัดสิน: ระบบแยก "เว้นว่าง = ยังไม่ประเมิน" (ไม่สร้างแถวคะแนน → ไปเข้าเงื่อนไข
  `complete:false` "ข้อมูลยังไม่ครบ") ออกจาก "0 = ประเมินแล้วได้ 0" ไว้แล้ว จึงไม่จำเป็นต้องกัน 0
  ไว้เป็นกรณีพิเศษ — และเดิมเด็กที่ได้ 1 มีระดับ แต่เด็กที่ได้ 0 ซึ่งต่ำกว่ากลับไม่มีระดับ เป็นผลกลับหัว
- พฤติกรรมเดิมก่อนแก้ (ตรวจแล้ว): `computeCombinedCompetencyResult()` คืน
  `complete:true, score:0, level:null, reason:"คะแนนไม่อยู่ในช่วงเกณฑ์แปลผลที่กำหนด"` →
  `summary.html` แสดง "0.0 / 100" พร้อมข้อความเตือนแทนชื่อระดับ (ไม่ crash แต่ครูอธิบายผลไม่ได้)

### จุดที่ต้องแก้ (ยังไม่ได้ลงมือ)

1. `schema.sql:1102` seed `'beginning'` เปลี่ยน `min_score` จาก `1` → `0`
2. **⚠ ต้องเพิ่ม migration `update` ต่างหาก** — seed ใช้ `on conflict (code) do nothing` และแถว
   `beginning` มีอยู่ในฐานข้อมูลจริงแล้ว **การแก้ seed เฉยๆ จะไม่มีผลกับ DB ที่รันไปแล้ว**:
   `update competency_interpretation_levels set min_score = 0 where code = 'beginning';`
3. `competency-manage.html:313` validation `rows[0].min_score!==1` → `!==0`
4. `competency-manage.html:127` ข้อความ "ช่วงต้องต่อกันตั้งแต่ 1–100" → "0–100"

**ไม่ต้องแก้**: `schema.sql:1097` constraint เขียน `min_score >= 0` อยู่แล้ว รองรับ 0 ได้ตั้งแต่ต้น ·
`supabase-client.js:167` ใช้ `find(l => score >= min && score <= max)` จะรับ 0 ได้เองทันทีที่ข้อมูล
เกณฑ์เปลี่ยน

## 2026-07-17 — ยกเลิกการแก้เกณฑ์ 0 (ทบทวนการตัดสินเดิม)

- **ผู้ใช้ทบทวนแล้วตัดสินใหม่: ไม่แก้** — ยกเลิกแผน 4 จุดในหัวข้อ "ตัดสินกติกา: คะแนนสมรรถนะรวม
  = 0" ด้านบน (ทั้ง seed, SQL update, validation และข้อความใน competency-manage.html)
- เหตุผล: ตรวจข้อมูลจริงบนเว็บแล้ว ไม่มีนักเรียนคนไหนมีคะแนนสมรรถนะรวม = 0 เคสนี้จึงยังไม่เคย
  ปรากฏจริง และการจะเกิดได้ต้องมีครูกรอกคะแนน 0 จริงครบทุกแหล่งที่มีน้ำหนัก (เว้นว่างไม่นับ เพราะ
  เข้าเงื่อนไข "ข้อมูลยังไม่ครบ" แยกไปแล้ว) — เป็น edge case ที่ยอมรับได้ ถ้าเกิดจริงจะเห็นข้อความ
  "คะแนนไม่อยู่ในช่วงเกณฑ์แปลผลที่กำหนด" ซึ่งไม่ใช่ข้อมูลผิด แค่ไม่มีชื่อระดับ ค่อยกลับมาแก้ตอนนั้นได้
- เกณฑ์กลางคงเดิม: 1–59 เริ่มต้น · 60–69 กำลังพัฒนา · 70–79 สามารถ · 80–100 เหนือความคาดหวัง

  (cosmetic — รายละเอียดในหัวข้อ "ข้อค้างที่พบเพิ่ม")

- ผู้ใช้ยืนยันให้ทำหน้า `competency-manage.html` เฉพาะ Admin: ฝั่งหน้าเว็บปฏิเสธบัญชีที่ไม่ใช่

    ภาคเรียนไว้ เหลือแก้ได้เฉพาะชื่อ เพื่อไม่ให้ snapshot และผลสรุปในอดีตคลาดเคลื่อน
  - ก่อนลบ ระบบนับองค์ประกอบ สมาชิก ครั้งประเมิน snapshot และคะแนนที่เกี่ยวข้อง; หากมีครั้ง
    ประเมินหรือคะแนน ต้องพิมพ์ชื่อรายการตรงกันก่อนจึงลบได้ (การลบเป็น cascade ตาม schema)
- ปรับ `schema.sql` ทั้ง policy หลักและ migration ท้ายไฟล์: เฉพาะ Admin สร้าง/แก้/ลบ
  `competency_assessments` และเขียน `competency_assessment_targets`/`members` ได้;
  **ไม่แตะ** policy ของ sessions/snapshot/scores จึงคงให้ครูเจ้าของกิจกรรมสร้างครั้งประเมินและ
  กรอกคะแนนใน `competency-entry.html` ได้เหมือนเดิม
- ตรวจผ่าน: parse module และ static assertions ของ Admin guard, ปุ่มแก้ไข/ลบ, การยืนยันลบ,
  และ RLS policy (`competency management access and action wiring: passed`)

  หรือสร้าง/ลบรายการทดสอบใด ๆ
