# CLAUDE.md — ระบบเก็บคะแนนหลักสูตรฐานสมรรถนะ (CRS Score)

ไฟล์นี้คือบริบทของโปรเจกต์ อ่านก่อนเริ่มงานทุกครั้ง

## ภาพรวม

เว็บแอปสำหรับครูกรอกคะแนนสมรรถนะนักเรียน (ประถม + มัธยม) แบบ mobile-first
สแตก: HTML/CSS/JS ธรรมดา (ไม่ใช้ framework) + Supabase (Postgres) + deploy บน GitHub Pages
สถานะ: อยู่ในช่วง **ทดสอบ workflow** ไม่ใช่ระบบสมบูรณ์ เป้าหมายคือให้ครูทดลองกรอกจริงก่อน
แล้วค่อยเพิ่มฟีเจอร์ทีละส่วน — อย่าพยายามสร้างให้ครบทุกอย่างในคราวเดียว

## สถานะปัจจุบัน

ทำเสร็จแล้ว — ครบ 4 หน้าหลัก deploy อยู่บน GitHub Pages ใช้งานได้จริงบนมือถือ:

## Architecture ส่วนคะแนน — ล็อกแล้ว (schema.sql คือความจริงหลัก)

โครงสร้าง 4 ชั้น ใช้ร่วมกันทั้ง "คะแนนวิชา" (`units.kind = 'วิชา'`) และ
"สมรรถนะหลัก 6 ด้าน" (`units.kind = 'สมรรถนะหลัก'`):

```
subjects → units → indicators → collections → scores
```

`scores` เก็บคะแนนดิบต่อ (นักเรียน × collection) เท่านั้น ทุกชั้นบนนั้นคำนวณสดตอนแสดงผล

กติกาคะแนน:
- ครูกำหนด `max_score` ได้อิสระทุกชั้น ยกเว้นสมรรถนะหลัก 6 ด้าน ที่ `max_score = 100` ตายตัวเสมอ
- คะแนนดิบรวมของชั้นล่างไม่จำเป็นต้องเท่ากับ `max_score` ของชั้นบน — เทียบสัดส่วน
  (บัญญัติไตรยางศ์) ให้อัตโนมัติตอนแสดงผล ไม่เก็บค่าที่เทียบแล้วลง database
- **สมรรถนะหลัก 6 ด้าน: แสดงแยกด้าน ด้านละคะแนนเต็ม 100 ไม่ต้องรวมเป็นเลขเดียว** (เคลียร์แล้ว)

### สูตรเทียบสัดส่วนคะแนน (ไล่ล่างขึ้นบน ต่อนักเรียน 1 คน)

```
indicator_raw    = SUM(scores.raw_score) ของทุก collection ใน indicator นี้
indicator_cap    = SUM(collections.max_score) ของทุก collection ใน indicator นี้
indicator_scaled = (indicator_raw / indicator_cap) * indicators.max_score   -- cap=0 → 0

unit_raw    = SUM(indicator_scaled) ของทุก indicator ใน unit นี้
unit_cap    = SUM(indicators.max_score) ของทุก indicator ใน unit นี้
unit_scaled = (unit_raw / unit_cap) * units.max_score

subject_raw    = SUM(unit_scaled) ของทุก unit ที่ kind='วิชา'
subject_cap    = SUM(units.max_score) ของทุก unit ที่ kind='วิชา'
subject_scaled = (subject_raw / subject_cap) * subjects.max_score
```

สมรรถนะหลัก 6 ด้าน (`units.kind = 'สมรรถนะหลัก'`): ใช้สูตร `unit_scaled` เดียวกัน
แสดงทีละด้าน (เต็ม 100) ไม่ต้องรวมต่อ

> **บริบทจากผู้ใช้**: คะแนนสมรรถนะหลักของนักเรียนจริงๆ มาจาก 3 แหล่ง — รายวิชา, กิจกรรม,
> และกิจวัตรประจำวันในโรงเรียน **ตอนนี้ระบบรองรับแค่แหล่งเดียวคือรายวิชา** (กรอกผ่าน
> `index.html` ผูกกับ `subject_id`) กิจกรรมกับกิจวัตรประจำวันยังไม่มีที่เก็บข้อมูล/หน้ากรอก
> เป็นงานถัดไป (ยังไม่ต้องออกแบบตอนนี้ รอผู้ใช้สั่งเพิ่ม) — เมื่อเพิ่ม 2 แหล่งนี้ ต้องคิด
> โครงสร้างใหม่ที่ไม่ผูกกับ `subject_id` อย่างเดียว (กิจกรรม/กิจวัตรไม่ใช่วิชา) และหน้าสรุปคะแนน
> ต้องรวมทั้ง 3 แหล่งเข้าด้วยกัน — ดู `summary.html` ส่วนวิชาบูรณาการที่เลือกโชว์สมรรถนะหลัก
> **แยกตามแหล่งที่มา ไม่ถัวเฉลี่ยรวม** ไว้แล้ว (เพราะแหล่งข้อมูลมักไม่ครบทุกแหล่งพร้อมกัน
> ถัวเฉลี่ยรวมจะทำให้คะแนนต่ำลงผิดๆ) — หลักการนี้ควรใช้ต่อเนื่องเมื่อเพิ่ม 2 แหล่งใหม่ด้วย

### ข้อมูลวิชา (`subjects`) เพิ่มเติม — implement แล้ว

```
code           รหัสวิชา เช่น ว21101 — ทุกวิชาควรมี (บังคับกรอกฝั่งหน้าเว็บ manage.html)
credits        หน่วยกิต — ใช้เฉพาะ level='มัธยม' (ประถมไม่มีหน่วยกิตแบบมัธยม) nullable เสมอ
```

## Architecture ส่วนวิชาบูรณาการ — implement แล้ว (schema.sql + manage.html + summary.html)

โครงสร้างวิชาทั้งประถมและมัธยม มี 2 ประเภท (`subjects.subject_type`):
- **วิชาพื้นฐาน** (`'พื้นฐาน'`) — วิชาเดี่ยว มีหน่วยใหญ่/หน่วยย่อย/ครั้งที่ของตัวเอง กรอกคะแนน
  ตรงที่ `index.html` เหมือนเดิมทุกอย่าง
- **วิชาบูรณาการ** (`'บูรณาการ'`) — วิชาที่รวมวิชาพื้นฐานหลายวิชาเข้าด้วยกัน (เช่น วิชาบูรณาการ
  "ผู้ประกอบการ" รวม สังคม + การงาน + ศิลปะ + วิทยาการคำนวณ) **ไม่มีหน่วยใหญ่/สมรรถนะหลักเป็น
  ของตัวเอง และ ไม่กรอกคะแนนตรง** — ครูยังกรอกคะแนน/เช็คชื่อแยกที่วิชาพื้นฐานแต่ละตัวตามปกติ
  ที่ `index.html`/`attendance.html` (dropdown วิชาทั้งสองหน้านี้กรองโชว์เฉพาะวิชาพื้นฐาน)

ตาราง `integration_members` (subject_id ทั้งสองฝั่งชี้ไปที่ `subjects`):
```
integrated_subject_id   วิชาบูรณาการ
member_subject_id       วิชาพื้นฐานที่เป็นสมาชิก (ต้องเป็น subject_type='พื้นฐาน')
unique(integrated_subject_id, member_subject_id)
```
จัดการสมาชิกได้ที่ `manage.html` (เลือกวิชาบูรณาการ → เพิ่ม/เอาวิชาพื้นฐานออก)

### สูตรคะแนน + ผลการเรียนของวิชาบูรณาการ (คำนวณตอนแสดงผลที่ `summary.html` ไม่ store)

```
1. คำนวณผลของวิชาพื้นฐานแต่ละตัวที่เป็นสมาชิก ตามสูตร subject_scaled + ผลการเรียน (ร./มส./เกรด)
   ข้างบนทุกอย่างตามปกติ (แยกอิสระต่อวิชา)

2. เช็ค มส./ร. ของวิชาบูรณาการ จากผลของวิชาพื้นฐานสมาชิกทั้งหมด:
   - ถ้ามีวิชาพื้นฐานอย่างน้อย 1 วิชาเป็น "มส." → วิชาบูรณาการ = "มส." ทันที
     (มส. ชนะ ร. เสมอเมื่อขัดแย้งกัน — ยืนยันแล้ว)
   - ถ้าไม่มี มส. แต่มีวิชาพื้นฐานอย่างน้อย 1 วิชาเป็น "ร." → วิชาบูรณาการ = "ร."
   - ถ้าไม่มีทั้งคู่ → ไปข้อ 3

3. คิดคะแนนวิชาบูรณาการ = ถัวเฉลี่ยถ่วงน้ำหนัก percent_score ของวิชาพื้นฐานแต่ละตัว
   น้ำหนัก = subjects.total_periods ของวิชาพื้นฐานนั้น (ยืนยันแล้ว: ใช้เวลาเรียน ไม่ใช่หน่วยกิต)

   integrated_percent = SUM(member.percent_score * member.total_periods)
                       / SUM(member.total_periods)    -- ถ้า SUM น้ำหนัก = 0 (ไม่มีวิชาไหนตั้ง
                                                          total_periods เลย) ได้ 0 ไปเลย เตือน
                                                          ผู้ใช้ในหน้าจอ

   แล้วแปลง integrated_percent เป็นเกรดด้วยเกณฑ์ 8 ระดับเดียวกับวิชาพื้นฐาน (ดูสูตรผลการเรียน)
```

> **ยืนยันแล้ว**: น้ำหนักถัวเฉลี่ย = เวลาเรียน (`total_periods`) ไม่ใช่หน่วยกิต (`credits`)
> เพราะประถมไม่มีหน่วยกิต ใช้เวลาเรียนจะครอบคลุมทั้งสองระดับชั้นได้ / ลำดับ ร.-มส. ที่ขัดแย้งกัน
> ให้ มส. ชนะเสมอ — ทั้งสองข้อ confirm กับผู้ใช้แล้วเมื่อตอนออกแบบฟีเจอร์นี้

## Architecture ส่วนเช็คชื่อ + ผลการเรียน — implement แล้ว (schema.sql + attendance.html + summary.html)

### ตารางใหม่ที่ต้องเพิ่ม

```
attendance_sessions   ครั้งที่เช็คชื่อ
  subject_id, session_date, periods_covered
  (periods_covered = จำนวนคาบที่เช็คครั้งนี้ครอบคลุม ครูกำหนดเองทุกครั้ง ไม่ fix)

attendance_records    ผลเช็คชื่อต่อคนต่อครั้ง
  session_id, student_id, status
  status ∈ {'มา' (default), 'ขาด', 'ลาป่วย', 'ลากิจ', 'มาสาย'}
  unique(session_id, student_id)

remarks                สถานะพิเศษที่ครูติดเอง (ตอนนี้ใช้แค่ ร.)
  student_id, subject_id, code ('ร.'), reason (text, บังคับกรอกเสมอ — ครูต้องระบุเหตุผล), created_at
```

เพิ่มคอลัมน์ใน `subjects`:
```
total_periods   จำนวนคาบเรียนทั้งหมดทั้งเทอมของวิชานี้ (ครูกำหนดตอนสร้างวิชา)
                ใช้เป็นตัวหารคำนวณเปอร์เซ็นต์เวลาเรียนสำหรับ มส.
```

### สูตรผลการเรียน (ต่อนักเรียน ต่อวิชา คำนวณตอนแสดงผล ไม่ store)

```
1. เช็ค remarks ก่อน: มีแถว code='ร.' ของ (student, subject) นี้ไหม
   → มี: แสดง "ร." พร้อม reason ที่ครูกรอกไว้ หยุดตรงนี้ ไม่ต้องคำนวณเกรด

2. ถ้าไม่ติด ร. (กฎ มส. ใช้ทั้งประถมและมัธยม — เปลี่ยนจากเดิมที่ใช้เฉพาะมัธยม ยืนยันแล้ว)
   คำนวณ % เวลาเรียน**ดิบ** (ยังไม่รวมชั่วโมงชดเชย) ก่อน:
   raw_attended_periods = SUM(periods_covered) ของ session ที่ status = 'มา' หรือ 'มาสาย'   -- น้ำหนักเต็ม 1.0
                         + SUM(periods_covered) ของ session ที่ status ∈ ('ลาป่วย','ลากิจ') * 0.5  -- ลา 2 ครั้ง = ขาด 1 ครั้ง
                         -- 'ขาด' ไม่นับเลย (น้ำหนัก 0)
   raw_percent_attend = raw_attended_periods / subjects.total_periods * 100

   มส. แบ่ง 2 ระดับตาม raw_percent_attend (ยืนยันกับผู้ใช้แล้ว 2026-07):
   - raw_percent_attend < 60  → "มส. (เรียนซ้ำรายวิชา)" หยุดตรงนี้เลย **ห้ามใช้ชั่วโมง
     ชดเชยกู้คืนเด็ดขาด** ไม่ว่า makeup_hours จะมีเท่าไหร่ก็ตาม — ต้องลงทะเบียนเรียนใหม่
     ทั้งวิชา (ระบบไม่มีเวิร์กโฟลว์ "เรียนซ้ำ" อัตโนมัติ ครูจัดการนอกระบบ/สร้าง enrollment
     ใหม่เอง)
   - 60 <= raw_percent_attend < 80  → "มส. (เรียนเพิ่มเติมให้ครบเวลา)" ตรงนี้เท่านั้นที่ใช้
     ชั่วโมงชดเชยได้:
       percent_attend_with_makeup = (raw_attended_periods + SUM(makeup_hours.periods)
                                      ของ student,subject นี้) / subjects.total_periods * 100
       ถ้ายังไม่ถึง 80% → ยัง "มส. (เรียนเพิ่มเติม)" อยู่ (โชว์ชดเชยไปแล้วกี่คาบ)
       ถ้าถึง 80% แล้ว → หลุด มส. ไปคิดเกรดต่อข้อ 3 (โชว์โน้ตว่าใช้ชดเชยไปกี่คาบ)
   - raw_percent_attend >= 80  → ไม่ติด มส. เลย ไปคิดเกรดต่อข้อ 3 ปกติ

3. ถ้าไม่ติดทั้ง ร. และ มส. → แปลงคะแนนเป็นเกรด (เกณฑ์มาตรฐาน 8 ระดับ):
   percent_score = subject_scaled / subjects.max_score * 100
   >= 80 → 4      >= 65 → 2.5     >= 50 → 1
   >= 75 → 3.5    >= 60 → 2       <  50 → 0
   >= 70 → 3      >= 55 → 1.5
```

> **ยืนยันแล้ว** (เช็คกับระเบียบกระทรวงศึกษาธิการ ว่าด้วยการวัดและประเมินผลการเรียนรู้
> ตามหลักสูตรแกนกลางการศึกษาขั้นพื้นฐาน พ.ศ. 2551 ใช้เหมือนกันทั้งโรงเรียนสังกัด สพฐ.
> และ สช. — ไม่มีเกณฑ์ สช. แยกต่างหาก): มาสาย = นับเป็นเวลาเรียนเต็ม, ขาด = ไม่นับเลย,
> ลาป่วย/ลากิจ = นับครึ่งเวลา (2 ครั้ง = ขาด 1 ครั้ง ตามที่โรงเรียนกำหนดเพิ่มเติมจากระเบียบ
> เพื่อความยืดหยุ่น — ระเบียบกลางไม่ได้ระบุอัตราส่วนนี้ไว้ตายตัว)
>
> **"ผ่อนผัน" แบบข้ามเช็คทั้งหมด — ยังไม่ implement และผู้ใช้เลือกไม่ทำ**: คุยกันแล้วว่า
> แทนที่จะข้ามเช็ค มส. ทั้งหมดแบบผ่อนผัน (ดุลยพินิจสถานศึกษาแบบเหมารวม) ผู้ใช้อยากได้กลไก
> "ชดเชยชั่วโมง" มากกว่า — ให้นักเรียนชดเชยเวลาที่ขาด (ทำงาน/เรียนเสริม ฯลฯ) แล้วครูบันทึก
> จำนวนคาบที่ชดเชยได้ (`makeup_hours` — implement แล้ว) ระบบจะ**บวกเข้า attended_periods
> ตรงๆ** ไม่ใช่ข้ามการเช็ค กลไก "ผ่อนผัน" แบบเดิม (remarks.code='ผ่อนผัน') ยังไม่ได้ทำ
> เก็บไว้เป็นตัวเลือกสำรองถ้ามีเคสที่ต้องข้ามเช็คจริงๆ ในอนาคต (ยังไม่ confirm ว่าจะทำเมื่อไหร่)
>
> **มส. 2 ระดับ — ยืนยันกับผู้ใช้แล้ว 2026-07**: ไม่ใช่แค่ "ถึง 80% ไหม" เฉยๆ แต่แบ่งตาม
> raw_percent_attend (ก่อนหักชั่วโมงชดเชย) เป็น 60-79.99% (เรียนเพิ่มเติม ใช้ชั่วโมงชดเชย
> กู้คืนได้) กับ ต่ำกว่า 60% (เรียนซ้ำรายวิชา ชั่วโมงชดเชยกู้คืนไม่ได้เด็ดขาด) —
> `attendance.html` บล็อกการบันทึกชั่วโมงชดเชยตั้งแต่ต้นถ้า raw < 60% (เช็คด้วย
> `getRawAttendPercent()`) และ `summary.html` ก็ไม่บวก makeup เข้า raw ให้เลยถ้า raw < 60%
> เป็นการล็อก 2 ชั้น (กัน UI บวกกันเผื่อ RLS/validation ฝั่งอื่นหลุด)

## Architecture ส่วน Auth + RLS — implement ครบแล้วทั้ง 2 เฟส (ทดสอบผ่านแล้ว)

ระบบล็อกอิน: อีเมล + รหัสผ่าน ผ่าน Supabase Auth (ยืนยันกับผู้ใช้แล้ว — ไม่ใช้ magic link/Google)
2 role: **admin** (แก้/ลบได้ทุกวิชา) และ **teacher** (เห็นทุกวิชา แต่แก้/ลบได้เฉพาะวิชาตัวเอง)
— นักเรียน (`students`) แก้ได้เฉพาะ admin ไปก่อน (ยังไม่มีหน้าจัดการนักเรียน)

**การสร้างบัญชี**: admin สร้างเองผ่าน Supabase Dashboard (Authentication > Add user) ไม่มี
หน้าสมัครสมาชิกบนเว็บ (ตัดสินใจแล้ว — เหมาะกับกลุ่มครูเล็กในโรงเรียนเดียว ง่ายกว่ามาก)
บัญชีใหม่ทุกบัญชีได้ role='teacher' อัตโนมัติผ่าน trigger (`handle_new_user`) ต้องรัน SQL
`update profiles set role='admin' where email='...'` เองถึงจะเลื่อนเป็น admin

### ไฟล์ที่เกี่ยวข้อง
```
supabase-client.js   ตั้งค่า Supabase client กลาง + requireAuth()/signOut()/getProfile()
                      ทุกหน้า (index/attendance/summary/manage) import จากไฟล์นี้แทนที่จะ
                      สร้าง client เองแยกกัน (เดิมมี URL/KEY hardcode ซ้ำ 4 ที่ ตอนนี้ที่เดียว)
login.html            หน้าล็อกอิน เรียก sb.auth.signInWithPassword() แล้วเด้งไป ?next= หรือ
                      index.html — ถ้าล็อกอินอยู่แล้วข้ามหน้านี้ไปเลย
```
ทุกหน้าที่ต้องล็อกอินก่อนใช้ เรียก `const session = await requireAuth();` เป็นบรรทัดแรกของ
สคริปต์ — ถ้าไม่มี session จะเด้งไป `login.html?next=<หน้าปัจจุบัน>` อัตโนมัติ (ด่านหน้า
ฝั่ง frontend) ส่วนด่านหลังที่คุมจริงคือ RLS ที่ฐานข้อมูล (เปิดแล้ว — ดูด้านล่าง)

### ตาราง/ฟังก์ชันที่เพิ่ม (schema.sql เฟส 1)
```
profiles            id (=auth.users.id), email, name, role ('admin'/'teacher' default teacher)
subjects.owner_id   ครู/บัญชีที่เป็นเจ้าของวิชานั้น (uuid → auth.users) nullable
                     (วิชาเก่าก่อนมี auth ยังไม่มีเจ้าของ — admin แก้ได้อย่างเดียวจนกว่าจะ
                     ตั้งเจ้าของให้ผ่าน manage.html ช่อง "เจ้าของวิชา" เห็นเฉพาะ admin)

is_admin()                      -- true ถ้า role ของคนที่ล็อกอินอยู่ = admin
can_edit_subject(subject_id)    -- is_admin() หรือ owner_id ตรงกับคนล็อกอินอยู่
can_edit_unit/indicator/collection/session(...)  -- ไล่ join กลับไปเช็ค can_edit_subject
```
ทุกฟังก์ชัน `security definer` เพื่อไม่ให้ชนกับ RLS ของตารางที่มันอ่านเอง (pattern มาตรฐาน
ของ Supabase สำหรับ helper function ใน RLS policy)

### กติกา RLS (เปิดใช้งานจริงแล้ว — `enable row level security` รันแล้วทุกตาราง)
ทุกตาราง: **อ่านได้ถ้าล็อกอินแล้ว** (`auth.role() = 'authenticated'`) ไม่ว่าจะเป็นเจ้าของหรือ
ไม่ก็ตาม (ครูเห็นวิชาคนอื่นได้ ยืนยันแล้ว) — **แก้ไข/ลบ/เพิ่ม** ต้องผ่าน `can_edit_subject`
(หรือฟังก์ชันลูกที่ไล่กลับไปหา subject เดียวกัน) ยกเว้น `students` ที่ `is_admin()` เท่านั้น
ทดสอบแล้ว (2026-07-12): request ที่ไม่มี session (แค่ anon key) อ่าน `subjects` ได้ `[]`
และ PATCH ไม่มีผลอะไรเลย — ปิดช่องโหว่ "ใครมีลิงก์ก็แก้ข้อมูลได้" สำเร็จ

> **ทางฉุกเฉิน**: ถ้า RLS ทำเว็บพัง มีคำสั่ง `alter table X disable row level security;`
> ของทุกตาราง comment ไว้ท้ายสุดของ schema.sql พร้อมรันย้อนกลับได้ทันที

### ขั้นตอนเพิ่มบัญชีครูใหม่ (operational — ใช้ทุกครั้งที่มีครูคนใหม่เข้าระบบ)
```
1. Supabase Dashboard > Authentication > Add user — กรอกอีเมล+รหัสผ่านให้ครู (auto confirm)
   (trigger handle_new_user จะสร้างแถว profiles ให้อัตโนมัติ role='teacher' ทันที)
2. เข้า manage.html (ด้วยบัญชี admin) > เลือกวิชาที่ครูคนนั้นสอนทีละวิชา > ช่อง
   "เจ้าของวิชา" (เห็นเฉพาะ admin) > เลือกชื่อครูคนนั้น > บันทึก — ทำซ้ำทุกวิชาที่สอน
3. ครูคนนั้นล็อกอินที่ login.html ได้เลย เห็นทุกวิชา แต่แก้/ลบได้เฉพาะวิชาที่ตั้งเป็นเจ้าของ
   ในข้อ 2 (ถ้ายังไม่ตั้งเจ้าของ จะเห็นได้แต่แก้ไม่ได้ — ปุ่มแก้ไข/ลบจะซ่อนอัตโนมัติ)
```
เลื่อนใครเป็น admin เพิ่ม: `update profiles set role = 'admin' where email = '...';`

## Architecture ส่วนนักเรียน + การลงทะเบียน — implement แล้ว

`students` เพิ่ม `grade_level` (คู่กับ `classroom` เดิม) — สองคอลัมน์นี้ใช้จับคู่ตอน
"เพิ่มทั้งห้อง" ที่ manage.html เท่านั้น ไม่มีผลอย่างอื่น

ตาราง `enrollments(subject_id, student_id)` many-to-many คือความจริงหลักว่า "วิชานี้มีใคร
เรียนบ้าง" — ทุกหน้าที่เคยดึง "นักเรียนทั้งหมด" (index/attendance/summary) เปลี่ยนมาเรียก
`getRosterForSubject(subjectId)` แทน (อยู่ใน supabase-client.js) ซึ่ง query จาก enrollments
ล้วนๆ ไม่ query จาก grade_level/classroom ตรงๆ — การจับคู่ชั้น/ห้องเป็นแค่ **ทางลัดตอนเพิ่ม
ข้อมูล** ผ่านปุ่ม "เพิ่มทั้งห้อง" (bulk insert เข้า enrollments) ไม่ใช่กฎที่บังคับใช้ตลอด
เพราะงั้นเพิ่ม/ลบนักเรียนออกจากวิชาทีละคนได้อิสระ ไม่ผูกกับห้องจริงเป๊ะๆ (รองรับเคสนักเรียน
ซ้ำชั้น/ติด มส. ที่ต้องเรียนร่วมห้องอื่น — ยืนยันกับผู้ใช้แล้ว)

`summary.html` วิชาบูรณาการ: รายชื่อนักเรียน = รวมนักเรียนที่ลงทะเบียนในวิชาพื้นฐานสมาชิก
**ทุกตัว** (union, ไม่ซ้ำคน) ไม่ได้มี enrollments ของตัวเองแยกต่างหาก เพราะวิชาบูรณาการไม่มี
คะแนน/เช็คชื่อเป็นของตัวเองอยู่แล้ว

สิทธิ์: อ่าน enrollments ได้ทุกคนที่ล็อกอิน แก้ไข (เพิ่ม/ลบ) ได้เฉพาะ `can_edit_subject`
เดียวกับที่ใช้กับ units/scores ทั้งหมด (เจ้าของวิชาหรือ admin) — `students` เองยังเขียนได้
เฉพาะ admin (ตัดสินใจเดิม ยังไม่เปลี่ยน)

## งานถัดไป (ยังไม่ได้เริ่ม — เลือกได้ว่าจะทำอันไหนก่อน)

ไม่มีลำดับบังคับแล้ว (แผนเดิม + วิชาบูรณาการเสร็จหมดแล้ว) ตัวเลือกที่เหลือ เรียงตามที่คาดว่า

   ไปเติมให้ครบ (owner_id ว่าง = แก้ได้แค่ admin คนเดียวตอนนี้)
2. **แหล่งคะแนนสมรรถนะหลักเพิ่ม 2 แหล่ง**: กิจกรรม, กิจวัตรประจำวัน (ตอนนี้มีแค่รายวิชา) —
   ผู้ใช้บอกไว้ว่าจะทำทีหลัง ยังไม่ต้องออกแบบตอนนี้ (ดูหมายเหตุใน Architecture ส่วนคะแนน)
3. "ผ่อนผัน" แบบข้ามเช็ค มส. ทั้งหมด — ไม่ใช่แผนตอนนี้ (เลือกทำ "ชั่วโมงชดเชย" แทนแล้ว)
   เก็บไว้เป็นตัวเลือกสำรอง ยังไม่ confirm ว่าจะทำเมื่อไหร่

**สไตล์หน้า**: ต่อยอดจาก `index.html`/`attendance.html`/`summary.html`/`manage.html`/
`students.html` เดิม — mobile-first, ฟอนต์ Sarabun, ธีมสี `--teal` เดียวกัน, ไม่ใช้ framework,

## ข้อควรระวัง

- **Supabase free tier**: โปรเจกต์ auto-pause ถ้าไม่มีการใช้งานติดต่อกัน 7 วัน ต้องเข้า
  dashboard กด resume เอง ไม่มี auto-backup
- **RLS เปิดใช้งานจริงแล้ว** (ทั้งเฟส 1 และเฟส 2) ทดสอบแล้วว่าปิดช่องโหว่ anon เข้าถึงข้อมูล
  ได้จริง — ถ้าจะแก้ policy หรือ schema เพิ่มทีหลัง ระวังอย่าทำให้ policy เดิมหลุด/พังโดยไม่ตั้งใจ
  เตือนผู้ใช้ทุกครั้งก่อนแตะส่วนนี้ (ความเสี่ยง: ล็อกผู้ใช้จริงออกจากข้อมูลตัวเอง)
- **repo เป็น public** เพื่อใช้ GitHub Pages ฟรี — ใช้ได้แค่ Supabase publishable key
  (`sb_publishable_...`) ในโค้ดฝั่งนี้เท่านั้น ห้ามใส่ secret key เด็ดขาด
- **ข้อมูลวิชาเป็นของจริงแล้ว** (ไม่ใช่ seed อีกต่อไป) ผู้ใช้กรอกวิชาจริงไปแล้ว 25+ วิชา
  (ม.3 หลายเทอม มีทั้งวิชาพื้นฐานและบูรณาการ) — นักเรียน 4 คนที่เหลือยังเป็นข้อมูลตัวอย่างอยู่
  (รอหน้าจัดการนักเรียน) ระวังอย่าลบ/แก้ข้อมูลวิชาจริงโดยไม่ได้รับอนุญาตชัดเจน
- สูตร มส. ยืนยันแล้ว (อิงระเบียบ ศธ. + กติกาลา 2:1 ที่โรงเรียนกำหนดเพิ่ม) ใช้ทั้งประถมและมัธยม
  — "ผ่อนผัน" ยังไม่ implement

## หลักการทำงานร่วมกัน

- เป้าหมายตอนนี้คือทดสอบ workflow ให้ครูใช้จริงได้ก่อน ไม่ใช่สร้างครบทุกฟีเจอร์รอบเดียว
- คอมเมนต์ในโค้ดเป็นภาษาไทย ชื่อตาราง/ตัวแปรเป็นภาษาอังกฤษ ตามที่ schema.sql วางไว้
- แก้ไฟล์ใหญ่ ให้ส่งไฟล์เต็มแทนการทำ diff เล็กๆ (ลด regression risk)
- ล็อกการตัดสินใจแล้วอย่าเปิดถกใหม่ ถ้าจะเปลี่ยนต้องมีเหตุผลใหม่จริงๆ ไม่ใช่แค่ความชอบ
- ตัวเลข/เกณฑ์ที่กระทบเกรดจริงของนักเรียน (เกณฑ์เกรด, เกณฑ์ มส., นิยาม ร.) ห้ามเดาเอง
  ต้องถามผู้ใช้ยืนยันก่อนเสมอ
