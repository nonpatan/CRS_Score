# AGENTS.md — ระบบเก็บคะแนนหลักสูตรฐานสมรรถนะ (CRS Score)

ไฟล์นี้คือบริบทของโปรเจกต์ อ่านก่อนเริ่มงานทุกครั้ง

## ภาพรวม

เว็บแอปสำหรับครูกรอกคะแนนสมรรถนะนักเรียน (ประถม + มัธยม) แบบ mobile-first
สแตก: HTML/CSS/JS ธรรมดา (ไม่ใช้ framework) + Supabase (Postgres) + deploy บน GitHub Pages
สถานะ: อยู่ในช่วง **ทดสอบ workflow** ไม่ใช่ระบบสมบูรณ์ เป้าหมายคือให้ครูทดลองกรอกจริงก่อน
แล้วค่อยเพิ่มฟีเจอร์ทีละส่วน — อย่าพยายามสร้างให้ครบทุกอย่างในคราวเดียว

## สถานะปัจจุบัน

ทำเสร็จแล้ว — ครบ 6 หน้าหลัก deploy อยู่บน GitHub Pages ใช้งานได้จริงบนมือถือ:
- ตารางคะแนน 6 ตัว (`subjects` → `units` → `indicators` → `collections` → `scores`, + `students`)
- ตารางเช็คชื่อ + ผลการเรียน (`attendance_sessions`, `attendance_records`, `remarks`,
  คอลัมน์ `subjects.total_periods`)
- `subjects` เพิ่มคอลัมน์ `code` (รหัสวิชา), `credits` (หน่วยกิต เฉพาะมัธยม), `grade_level`
  (ชั้น เช่น ป.1/ม.3), `subject_type` ('พื้นฐาน' หรือ 'บูรณาการ') + ตาราง `integration_members`
  (วิชาพื้นฐานที่ประกอบกันเป็นวิชาบูรณาการ 1 ตัว) — ดูหัวข้อ "Architecture ส่วนวิชาบูรณาการ"
- `entry.html` — กรอกคะแนน มีตัวเลือก "ประเภทคะแนน" สลับกรอกได้ทั้งคะแนนวิชา
  (`units.kind='วิชา'`) และสมรรถนะหลัก 6 ด้าน (`units.kind='สมรรถนะหลัก'`) — dropdown วิชา
  กรองเฉพาะ `subject_type='พื้นฐาน'` (วิชาบูรณาการกรอกคะแนนตรงไม่ได้) — ชื่อไฟล์เดิมคือ
  `index.html` ย้ายมาที่นี่แล้ว (ดูหัวข้อ "Architecture ส่วนหน้าแรก/ล็อกอิน")
- `attendance.html` — เช็คชื่อ (เลือกวิชา → วันที่ → จำนวนคาบ → ติ๊กสถานะรายคน
  แก้ไขครั้งที่เคยเช็คไปแล้วได้) — dropdown วิชากรองเฉพาะ `subject_type='พื้นฐาน'` เช่นกัน
- `summary.html` — สรุปคะแนน + ผลการเรียน เลือกวิชาพื้นฐานหรือวิชาบูรณาการก็ได้ → นักเรียนหรือ
  ทั้งห้อง → วิชาพื้นฐาน: คะแนนแต่ละหน่วยใหญ่ + รวมวิชา + สมรรถนะหลัก 6 ด้านแยกด้าน +
  ผลการเรียนเกรด/ร./มส. — วิชาบูรณาการ: ถัวเฉลี่ยถ่วงน้ำหนักจากวิชาพื้นฐานสมาชิก
- `manage.html` — จัดการโครงสร้าง ครูสร้าง/แก้/ลบวิชา (ทั้งพื้นฐานและบูรณาการ) + หน่วยใหญ่/
  หน่วยย่อย/ครั้งที่ (เฉพาะวิชาพื้นฐาน) + วิชาพื้นฐานสมาชิกของวิชาบูรณาการ เองผ่านเว็บได้แล้ว
  ไม่ต้องรัน SQL มือ — กันการลบถ้ามีคะแนนกรอกไว้แล้ว หรือยังเป็นสมาชิกวิชาบูรณาการอื่นอยู่

  ย่อยในแต่ละด้านยังเป็นชื่อ placeholder ("องค์ประกอบที่ 1/2") ไม่ใช่ของจริง
- `login.html` + `supabase-client.js` — ล็อกอินด้วยอีเมล/รหัสผ่าน + client กลางให้ทุกหน้าใช้ร่วมกัน
  ทุกหน้าที่ต้องล็อกอิน ต้องล็อกอินก่อนถึงจะเข้าได้ (เด้งไป login.html อัตโนมัติถ้ายังไม่ล็อกอิน)

- **`index.html` = ทางผ่านไป `login.html`** (ผู้ใช้ขอให้ login เป็นหน้าแรกเสมอเวลาเข้า URL ราก
  ของเว็บ) หน้ากรอกคะแนนตัวจริง (เดิมคือ `index.html`) ย้ายไปอยู่ **`entry.html`** แล้ว —
  ดูหัวข้อ "Architecture ส่วนหน้าแรก/ล็อกอิน"

  security` จริงทุกตาราง) รันบน Supabase แล้วทั้งคู่ ทดสอบแล้วว่า request ที่ไม่ล็อกอิน
  อ่าน/แก้ข้อมูลผ่าน API ตรงๆ ไม่ได้เลย (คืนค่าว่างเปล่า) มีบัญชี admin คนแรกแล้ว

- `students.html` — หน้าจัดการนักเรียน เพิ่ม/ลบทีละคนหรือวางจาก Excel ทีละหลายคน (5 คอลัมน์:

## Architecture ส่วนคะแนน — ล็อกแล้ว (schema.sql คือความจริงหลัก)

โครงสร้าง 4 ชั้น ใช้ร่วมกันทั้ง "คะแนนวิชา" (`units.kind = 'วิชา'`) และ
"สมรรถนะหลัก 6 ด้าน" (`units.kind = 'สมรรถนะหลัก'`):

```
subjects → units → indicators → collections → scores
```

`scores` เก็บคะแนนดิบต่อ (นักเรียน × collection) เท่านั้น ทุกชั้นบนนั้นคำนวณสดตอนแสดงผล

กติกาคะแนน:
- ครูกำหนด `max_score` ได้อิสระทุกชั้น ยกเว้นสมรรถนะหลัก 6 ด้าน ที่ `max_score = 100` ตายตัวเสมอ
- คะแนนดิบรวมของชั้นล่างไม่จำเป็นต้องเท่ากับ `max_score` ของชั้นบน — เทียบสัดส่วน
  (บัญญัติไตรยางศ์) ให้อัตโนมัติตอนแสดงผล ไม่เก็บค่าที่เทียบแล้วลง database
- **สมรรถนะหลัก 6 ด้าน: แสดงแยกด้าน ด้านละคะแนนเต็ม 100 ไม่ต้องรวมเป็นเลขเดียว** (เคลียร์แล้ว)

### สูตรเทียบสัดส่วนคะแนน (ไล่ล่างขึ้นบน ต่อนักเรียน 1 คน)

```
indicator_raw    = SUM(scores.raw_score) ของทุก collection ใน indicator นี้
indicator_cap    = SUM(collections.max_score) ของทุก collection ใน indicator นี้
indicator_scaled = (indicator_raw / indicator_cap) * indicators.max_score   -- cap=0 → 0

unit_raw    = SUM(indicator_scaled) ของทุก indicator ใน unit นี้
unit_cap    = SUM(indicators.max_score) ของทุก indicator ใน unit นี้
unit_scaled = (unit_raw / unit_cap) * units.max_score

subject_raw    = SUM(unit_scaled) ของทุก unit ที่ kind='วิชา'
subject_cap    = SUM(units.max_score) ของทุก unit ที่ kind='วิชา'
subject_scaled = (subject_raw / subject_cap) * subjects.max_score
```

สมรรถนะหลัก 6 ด้าน (`units.kind = 'สมรรถนะหลัก'`): ใช้สูตร `unit_scaled` เดียวกัน
แสดงทีละด้าน (เต็ม 100) ไม่ต้องรวมต่อ

> **บริบทจากผู้ใช้**: คะแนนสมรรถนะหลักของนักเรียนจริงๆ มาจาก 3 แหล่ง — รายวิชา, กิจกรรม,
> และกิจวัตรประจำวันในโรงเรียน ตอนนี้ระบบมีที่กำหนดและกรอกครบทั้ง 3 แหล่งแล้ว: รายวิชาใช้
> `entry.html`; กิจกรรม/กิจวัตรใช้ `competency-manage.html` และ `competency-entry.html`
> โดยเก็บแยกจาก `subject_id` — หน้าสรุปคะแนนยังไม่ได้รวมกิจกรรม/กิจวัตร และต้องคิดรูปแบบใหม่
> ต้องรวมทั้ง 3 แหล่งเข้าด้วยกัน — ดู `summary.html` ส่วนวิชาบูรณาการที่เลือกโชว์สมรรถนะหลัก
> **แยกตามแหล่งที่มา ไม่ถัวเฉลี่ยรวม** ไว้แล้ว (เพราะแหล่งข้อมูลมักไม่ครบทุกแหล่งพร้อมกัน
> ถัวเฉลี่ยรวมจะทำให้คะแนนต่ำลงผิดๆ) — หลักการนี้ควรใช้ต่อเนื่องเมื่อเพิ่ม 2 แหล่งใหม่ด้วย

## Architecture การแยกข้อมูลตามปีการศึกษา (year scoping) — implement แล้ว

**หลักการ**: `subjects.year` คือแกนหลักที่แยกข้อมูลข้ามปี — วิชา 1 แถวผูกกับปีเดียว (year+term+
grade_level) และทุกอย่างที่ห้อยกับวิชา (scores/attendance/remarks/makeup_hours/enrollments/
integration_members) จึงถูกผูกกับปีนั้นโดยอัตโนมัติผ่าน `subject_id` → "วิทยาการคำนวณ ม.3 ปี 2568"
กับ "...ปี 2569" เป็นคนละแถวกันสนิท ข้อมูลลูกไม่ปนกัน ปีเก่าไม่ถูกทับ

**ตัวเลือกปีกลาง (active year)** — ยืนยันกับผู้ใช้แล้ว: เลือกปีครั้งเดียว ทุกหน้าจำ:
- เก็บใน `localStorage` คีย์ `crs_active_year` ผ่าน `getActiveYear()`/`setActiveYear()` ใน
  `supabase-client.js` (+ `distinctYears(subjects)` รวบรวมปีที่มีจริงจากรายการวิชา)
- ทุกหน้าที่มี dropdown เลือกวิชา (`entry`/`attendance`/`summary`/`manage`) มี dropdown
  "ปีการศึกษา" (เติมจาก `distinctYears`) + ตัวเลือก "— ทุกปี —" — เปลี่ยนปีที่หน้าไหนก็ `setActiveYear`
  ให้หน้าอื่นจำด้วย — `applySubjectFilter()` กรองวิชาด้วย year (+ grade + term เหมือนเดิม)
- `subjectLabel()` ทุกหน้า **โชว์ปีเสมอ** (รหัส · ชื่อ · ปี 2568 · เทอม 1 · [บูรณาการ]) กันเลือกผิดปี
- `retention.html` sync กับปีกลางด้วย (เลือกชั้นแล้วถ้าปีกลางมีในชั้นนั้น จะเลือกให้อัตโนมัติ)
- `manage.html`: **บังคับกรอก year ตอนบันทึกวิชา** (เดิม nullable) + สร้างวิชาใหม่เติมปีกลางให้
  อัตโนมัติ + ตัวเลือกวิชาพื้นฐานสมาชิกของวิชาบูรณาการกรองด้วย year ด้วย (กันดึงวิชาข้ามปี)

> **ทำไมยังมี "— ทุกปี —"**: วิชาเก่าบางตัว (ที่ไม่ใช่ ม.3) `year` ยังว่าง (null) เพราะสร้างก่อน
> บังคับ field นี้ — ถ้ากรองปีตายตัววิชาพวกนั้นจะหาย เลยมี "ทุกปี" เป็นทางออก ค่อยๆ เก็บให้ครบ
> ทีหลัง (แก้วิชาที่ manage.html ทีไรจะโดนบังคับกรอกปีให้ครบเอง)

> **เรื่องนักเรียนข้ามปี**: ตาราง `students.grade_level`/`classroom` คงเป็นค่า **ปีปัจจุบัน**
> สำหรับหน้าปฏิบัติงานและถูกอัปเดตตอนเลื่อนชั้น แต่ประวัติจริงแยกเก็บที่
> `student_year_placements(student_id, year, grade_level, classroom)` แล้ว (PK UUID +
> `unique(student_id, year)`; ชั้นและห้องห้ามว่าง) — รายงานต้องใช้ placement ของปีที่เลือก
> จึงยังเห็นเด็กที่ปัจจุบันจบ/ย้ายออก และไม่สลับไปใช้ห้องของปีใหม่

## Architecture ส่วนขึ้นปีการศึกษาใหม่ / เลื่อนชั้น — implement แล้ว (`rollover.html`)

หน้า `rollover.html` (admin เท่านั้น) ยกของปีเก่าไปปีใหม่ ไม่ต้องสร้างใหม่หมด — ยืนยันโมเดลกับผู้ใช้แล้ว:

**Schema เพิ่ม (รันแล้ว):**
- `students.graduated boolean not null default false` — ธงจบการศึกษา (แยกออกจากรายชื่อ active)
- `students.left_school boolean not null default false` — ธงย้ายออก/ไม่เรียนต่อ (เช่น ป.6 ที่ไป
  เรียนมัธยมที่อื่น — ต่างจาก graduated ที่แปลว่าจบหลักสูตร) active = graduated=false และ left_school=false
- `student_year_placements` — ประวัติชั้น/ห้องรายปี (อ่านได้เมื่อ login, เขียนได้เฉพาะ admin)
  ใช้กับรายงานย้อนหลังและเป็นแหล่งข้อมูลชั้น/ห้องของปีต้นทางในหน้า rollover
- ตาราง `app_settings (key text pk, value text)` — ค่าตั้งค่าส่วนกลาง (RLS: authenticated อ่านได้,
  admin เขียน) เก็บ `highest_grade` = ชั้นสูงสุดที่โรงเรียนเปิดสอน (ตอนนี้ 'ม.3')

**Helpers ใน `supabase-client.js`:** `getSetting`/`setSetting`, `GRADE_ORDER` (ป.1→ม.6),
`nextGrade(grade, highest)` (คืน null = จบ ถ้า grade === highest), `promoteClassroom` (ม.2/1→ม.3/1),
`getActiveStudents()` (กรอง graduated=false)

**ขั้นตอนบนหน้า rollover (มีพรีวิว + ยืนยันก่อนกดจริง):**
0. ตั้งชั้นสูงสุดที่เปิดสอน (บันทึกลง app_settings) — ตัดสินว่าใครจบ
1. เลือกปีต้นทาง → ปีใหม่ (default = ต้นทาง+1)
2. เลื่อนชั้นนักเรียน: dropdown **3 ทางต่อคน** — เลื่อน (ชั้นสูงสุด=จบ) / คงชั้นเดิม(ซ้ำชั้น) /
   ย้ายออก(left_school=true) มีปุ่ม "ตั้งทั้งกลุ่ม" ต่อชั้น (เช่น ป.6 บางคนย้ายออก ตั้งกลุ่มก่อนแล้วแก้
   รายคน — ป.6 เรียนต่อ ม.1 ที่เดิมแค่ประมาณครึ่ง) + สรุปสดตาม dropdown
3. เลือกวิชาที่จะยก (default ทั้งหมด ติ๊กออกได้ถ้าจะปรับโครงสร้างใหม่)
4. กดรัน: **กันก๊อปซ้ำ** (ปีใหม่ต้องยังไม่มีวิชา) → ตรวจว่าปีต้นทางมี placement ครบ → ใช้ห้องของ
   placement ปีต้นทาง → เลื่อนชั้นและสร้าง placement ของปีใหม่ให้คนที่เรียนต่อ → ก๊อปวิชา+โครงสร้าง
   (พื้นฐานก่อน แล้ว remap สมาชิกบูรณาการ) คะแนน/เช็คชื่อว่างเปล่า → auto-enroll วิชาพื้นฐานปีใหม่
   จากเด็ก active ที่ชั้น+ห้องตรงกับปีต้นทาง → ตั้ง active year เป็นปีใหม่

> **ทำไม snapshot ห้องก่อนเลื่อนชั้น**: auto-enroll จับคู่ด้วย "ห้องที่วิชาต้นทางเคยมี" (เช่น ม.2/1)
> ซึ่งตรงกับป้ายห้องของเด็กที่เพิ่งเลื่อนขึ้นมาพอดี (ม.1/1→ม.2/1) — แต่ถ้าอ่านห้องหลังเลื่อนแล้ว
> เด็กเก่าย้ายไป ม.3/1 หมด จะจับคู่ผิด เลยต้องเก็บห้องไว้ก่อนเลื่อน

> **ทำไมก๊อปด้วย JS ไม่ใช้ Postgres function**: งานปีละครั้ง ข้อมูลน้อย ก๊อปทีละวิชาใน JS debug ง่าย
> กว่า + ไม่ต้องให้ผู้ใช้รัน plpgsql ยาวๆ — แลกกับความไม่ atomic (ถ้าพังกลางคันมี log บอกว่าถึงไหน
> ตัวกันก๊อปซ้ำจะบล็อกการรันซ้ำจนกว่าจะลบวิชาปีใหม่ที่ค้างออก)

> **graduated/left_school ใช้ที่ไหนบ้าง**: `students.html` (ปกติซ่อนคนจบ/ย้ายออก มี checkbox
> "แสดงคนที่ไม่ได้เรียนต่อ" + badge บอกเหตุ), `manage.html` enroll picker + `getActiveStudents`
> (กรองทั้ง graduated และ left_school — ไม่เสนอให้ลงทะเบียนใหม่) — แต่ `getRosterForSubject`
> **ไม่กรอง** เพราะเป็นรายชื่อต่อวิชา (ผูกปีแล้ว) คนจบ/ย้ายออกต้องยังเห็นในวิชาปีเก่าที่เคยเรียนเพื่อ
> ดูคะแนนย้อนหลัง

## Architecture ส่วนวิชาบูรณาการ — implement แล้ว (schema.sql + manage.html + summary.html)

โครงสร้างวิชาทั้งประถมและมัธยม มี 2 ประเภท (`subjects.subject_type`):
- **วิชาพื้นฐาน** (`'พื้นฐาน'`) — วิชาเดี่ยว มีหน่วยใหญ่/หน่วยย่อย/ครั้งที่ของตัวเอง กรอกคะแนน
  ตรงที่ `entry.html` เหมือนเดิมทุกอย่าง
- **วิชาบูรณาการ** (`'บูรณาการ'`) — วิชาที่รวมวิชาพื้นฐานหลายวิชาเข้าด้วยกัน (เช่น วิชาบูรณาการ
  "ผู้ประกอบการ" รวม สังคม + การงาน + ศิลปะ + วิทยาการคำนวณ) **ไม่มีหน่วยใหญ่/สมรรถนะหลักเป็น
  ของตัวเอง และ ไม่กรอกคะแนนตรง** — ครูยังกรอกคะแนน/เช็คชื่อแยกที่วิชาพื้นฐานแต่ละตัวตามปกติ
  ที่ `entry.html`/`attendance.html` (dropdown วิชาทั้งสองหน้านี้กรองโชว์เฉพาะวิชาพื้นฐาน)

ตาราง `integration_members` (subject_id ทั้งสองฝั่งชี้ไปที่ `subjects`):
```
integrated_subject_id   วิชาบูรณาการ
member_subject_id       วิชาพื้นฐานที่เป็นสมาชิก (ต้องเป็น subject_type='พื้นฐาน')
unique(integrated_subject_id, member_subject_id)
```
จัดการสมาชิกได้ที่ `manage.html` (เลือกวิชาบูรณาการ → เพิ่ม/เอาวิชาพื้นฐานออก)

> **แก้บั๊ก 2026-07 — ลบวิชาบูรณาการได้โดยไม่เช็คอะไรเลย**: วิชาบูรณาการเองไม่มีคะแนนของตัวเอง
> (คะแนนอยู่ที่วิชาพื้นฐานสมาชิก) เดิม guard ตอนลบเช็คเฉพาะ `subject_type === 'พื้นฐาน'` เท่านั้น
> วิชาบูรณาการเลย**ลบได้เลยไม่มีการเช็คอะไรเลย** ทั้งที่ลบแล้ว `integration_members` จะ cascade
> หายหมด (การจัดกลุ่มหาย แม้คะแนนจริงของวิชาพื้นฐานสมาชิกจะไม่ถูกลบก็ตาม — เสียแค่การตั้งค่า
> ไม่เสียคะแนน) ผู้ใช้เจอเคสจริงที่วิชาย่อยกรอกคะแนนไปแล้ว — แก้โดยไล่ดูวิชาพื้นฐานสมาชิกทุกตัว
> ก่อนลบ **แต่ตัดสินใจไม่บล็อก** (ยืนยันกับผู้ใช้แล้ว: ต่างจาก guard ฝั่งวิชาพื้นฐานที่บล็อกเลย
> เพราะลบวิชาพื้นฐานทำให้คะแนนจริงหาย ส่วนลบวิชาบูรณาการไม่ทำให้คะแนนของวิชาย่อยหายเลย —
> เสียแค่การจัดกลุ่ม) ถ้ามีวิชาย่อยตัวไหนมีคะแนนแล้ว จะ**เปลี่ยนข้อความยืนยันลบ** (`confirm()`)
> ให้บอกรายชื่อวิชาย่อย+จำนวนช่องคะแนนที่มี แล้วให้ผู้ใช้ตัดสินใจเองว่าจะลบต่อไหม (ไม่ใช่บล็อกเด็ดขาด)
>
> **แก้บั๊ก 2026-07 — หน้าเช็คชื่อไม่แสดงวิชาย่อยเมื่อเลือกวิชาบูรณาการ**: สาเหตุเกิดจากการ query ตาราง `integration_members` ใน `attendance.html` ที่ใช้คำสั่ง `.select("member:member_subject_id(*)")` ซึ่งขาด `*,` (ดึงคอลัมน์ของตารางหลัก) ทำให้ Supabase JS คืนค่าข้อมูลที่ฝังมา (embedded resources) ผิดพลาดในกรณีที่ตารางนี้มี Foreign Key ชี้ไปที่ `subjects` ถึง 2 ตัว — แก้ไขโดยเปลี่ยนเป็น `.select("*, member:member_subject_id(*)")` ให้ตรงกับหน้า `manage.html` และ `summary.html`

### สูตรคะแนน + ผลการเรียนของวิชาบูรณาการ (คำนวณตอนแสดงผลที่ `summary.html` ไม่ store)

```
1. คำนวณคะแนนของวิชาพื้นฐานแต่ละตัวที่เป็นสมาชิก ตามสูตร subject_scaled ปกติ (แยกอิสระต่อวิชา)
   — แต่ "ไม่เช็ค มส. รายวิชาย่อย" (computeSubjectResult เรียกด้วย skipMs=true) เพราะ มส.
   ของวิชาบูรณาการคิดแบบรวมในข้อ 2 แทน · ร. ยังเช็ครายวิชาย่อยเหมือนเดิม

2. เช็ค มส. "แบบรวม" ที่ระดับวิชาบูรณาการ (เปลี่ยนจากเดิมที่เช็คแยกรายวิชาย่อย — ยืนยันกับ
   ผู้ใช้แล้ว 2026-07 พร้อมตัวอย่าง: สังคม 20 + การงาน 20 = ฐานรวม 40 คาบ ต้องเข้าเรียน
   ไม่น้อยกว่า 80% คือ 32 คาบ โดยนับเวลาเรียนจากทุกวิชาย่อยรวมกัน):
   ฐานเวลารวม (totalBase)  = SUM(member.total_periods)
   ขาดสะสมรวม (rawMissed)  = SUM(คาบขาดของนักเรียนคนนี้ในทุกวิชาย่อย)   -- computeMissedPeriods ต่อวิชา
   ชดเชยรวม (makeupTotal)  = SUM(makeup_hours ของทุกวิชาย่อย)
   เพดาน: 20%/40% ของฐานรวม — ตรรกะ 2 ระดับ (เรียนเพิ่ม/เรียนซ้ำ) เหมือนวิชาพื้นฐานทุกอย่าง
   → ติด มส. รวม = วิชาบูรณาการติด มส. (มส. ชนะ ร. เสมอเมื่อขัดแย้งกัน — ยืนยันแล้ว)

   ถ้าไม่ติด มส. รวม แต่มีวิชาย่อยอย่างน้อย 1 วิชาติด "ร." → วิชาบูรณาการ = "ร."
   ไม่มีทั้งคู่ → ไปข้อ 3

3. คิดคะแนนวิชาบูรณาการ = ถัวเฉลี่ยถ่วงน้ำหนัก percent_score ของวิชาพื้นฐานแต่ละตัว
   น้ำหนัก = subjects.total_periods ของวิชาพื้นฐานนั้น (ยืนยันแล้ว: ใช้เวลาเรียน ไม่ใช่หน่วยกิต)

   integrated_percent = SUM(member.percent_score * member.total_periods)
                       / SUM(member.total_periods)    -- ถ้า SUM น้ำหนัก = 0 (ไม่มีวิชาไหนตั้ง
                                                          total_periods เลย) ได้ 0 ไปเลย เตือน
                                                          ผู้ใช้ในหน้าจอ

   แล้วแปลง integrated_percent เป็นเกรดด้วยเกณฑ์ 8 ระดับเดียวกับวิชาพื้นฐาน (ดูสูตรผลการเรียน)
```

> **ยืนยันแล้ว**: น้ำหนักถัวเฉลี่ย = เวลาเรียน (`total_periods`) ไม่ใช่หน่วยกิต (`credits`)
> เพราะประถมไม่มีหน่วยกิต ใช้เวลาเรียนจะครอบคลุมทั้งสองระดับชั้นได้ / ลำดับ ร.-มส. ที่ขัดแย้งกัน

### สูตรผลการเรียน (ต่อนักเรียน ต่อวิชา คำนวณตอนแสดงผล ไม่ store)

```
1. เช็ค remarks ก่อน: มีแถว code='ร.' ของ (student, subject) นี้ไหม
   → มี: แสดง "ร." พร้อม reason ที่ครูกรอกไว้ หยุดตรงนี้ ไม่ต้องคำนวณเกรด

2. ถ้าไม่ติด ร. (กฎ มส. ใช้ทั้งประถมและมัธยม — เปลี่ยนจากเดิมที่ใช้เฉพาะมัธยม ยืนยันแล้ว)
   คำนวณ **จำนวนคาบขาดสะสมดิบ** (ยังไม่หักชั่วโมงชดเชย) ก่อน — นับจาก session ที่เช็คชื่อ
   ไปแล้วเท่านั้น:
   raw_missed_periods = SUM(periods_covered) ของ session ที่ status = 'ขาด'   -- น้ำหนักเต็ม 1.0
                       + SUM(periods_covered) ของ session ที่ status ∈ ('ลาป่วย','ลากิจ') * 0.5  -- ลา 2 ครั้ง = ขาด 1 ครั้ง
                       -- 'มา'/'มาสาย' ไม่นับเลย (ไม่ขาด)

   **สำคัญ — ไม่ใช่ % ของคาบทั้งเทอม**: เดิมเคยเทียบเป็น `attended / total_periods * 100`
   ซึ่งพัง**ทันทีที่เปิดเทอมใหม่** (บั๊กที่ผู้ใช้เจอจริง 2026-07: เพิ่งเปิดเทอม เช็คชื่อไปแค่
   1-2 ครั้ง นักเรียนขาด 1-2 ครั้งนั้น ก็ติด มส. ทันทีทั้งที่ยังเหลือทั้งเทอมให้แก้ตัว เพราะ
   ตัวหารคือคาบ**ทั้งเทอม** แต่ตัวตั้งมีแค่คาบที่เช็คไปแล้วไม่กี่คาบ) แก้โดยเปลี่ยนมา **นับ
   จำนวนคาบขาดสะสมเป็นค่าสัมบูรณ์ เทียบกับเพดานคาบที่ขาดได้สูงสุด** (เพดานคำนวณจาก
   `total_periods` ทั้งเทอมเสมอ แต่ตัวขาดสะสมนับจากคาบที่เช็คจริงเท่านั้น — ต้นเทอมขาดสะสม
   ย่อมยังน้อยกว่าเพดานเยอะ ไม่ติด มส. ง่ายๆ) — ยืนยันกับผู้ใช้แล้ว 2026-07 ตัวอย่างที่ใช้
   ยืนยัน: วิชา 40 คาบ/เทอม ขาดได้สูงสุด 20% = 8 คาบ (เช่น ขาด 4 ครั้ง ครั้งละ 2 คาบ)

   max_missed_retake = subjects.total_periods * 0.40   -- ขาดเกินนี้ = เรียนซ้ำ (สมมูลกับเกณฑ์เดิม <60%)
   max_missed_makeup  = subjects.total_periods * 0.20   -- ขาดเกินนี้ = มส. เรียนเพิ่ม (สมมูลกับเกณฑ์เดิม <80%)

   - raw_missed_periods > max_missed_retake  → "มส. (เรียนซ้ำรายวิชา)" หยุดตรงนี้เลย **ห้ามใช้
     ชั่วโมงชดเชยกู้คืนเด็ดขาด** ไม่ว่า makeup_hours จะมีเท่าไหร่ก็ตาม — ต้องลงทะเบียนเรียนใหม่
     ทั้งวิชา (ระบบไม่มีเวิร์กโฟลว์ "เรียนซ้ำ" อัตโนมัติ ครูจัดการนอกระบบ/สร้าง enrollment
     ใหม่เอง)
   - max_missed_makeup < raw_missed_periods <= max_missed_retake  → "มส. (เรียนเพิ่มเติมให้ครบเวลา)"
     ตรงนี้เท่านั้นที่ใช้ชั่วโมงชดเชยได้ (ลบยอดขาดสุทธิตรงๆ ไม่ใช่บวกเข้า attended):
       net_missed = max(0, raw_missed_periods - SUM(makeup_hours.periods) ของ student,subject นี้)
       ถ้า net_missed > max_missed_makeup → ยัง "มส. (เรียนเพิ่มเติม)" อยู่ (โชว์ชดเชยไปแล้วกี่คาบ
       + ขาดสุทธิเหลือเท่าไหร่)
       ถ้า net_missed <= max_missed_makeup → หลุด มส. ไปคิดเกรดต่อข้อ 3 (โชว์โน้ตว่าใช้ชดเชยไปกี่คาบ)
   - raw_missed_periods <= max_missed_makeup  → ไม่ติด มส. เลย ไปคิดเกรดต่อข้อ 3 ปกติ

3. ถ้าไม่ติดทั้ง ร. และ มส. → แปลงคะแนนเป็นเกรด (เกณฑ์มาตรฐาน 8 ระดับ):
   percent_score = subject_scaled / subjects.max_score * 100
   >= 80 → 4      >= 65 → 2.5     >= 50 → 1
   >= 75 → 3.5    >= 60 → 2       <  50 → 0
   >= 70 → 3      >= 55 → 1.5
```

> ลาป่วย/ลากิจ = นับครึ่งเวลา (2 ครั้ง = ขาด 1 ครั้ง ตามที่โรงเรียนกำหนดเพิ่มเติมจากระเบียบ

## Architecture ส่วน Auth + RLS — implement ครบแล้วทั้ง 2 เฟส (ทดสอบผ่านแล้ว)

ระบบล็อกอิน: อีเมล + รหัสผ่าน ผ่าน Supabase Auth (ยืนยันกับผู้ใช้แล้ว — ไม่ใช้ magic link/Google)
2 role: **admin** (แก้/ลบได้ทุกวิชา) และ **teacher** (เห็นทุกวิชา แต่แก้/ลบได้เฉพาะวิชาตัวเอง)
— นักเรียน (`students`) แก้ได้เฉพาะ admin ไปก่อน (ยังไม่มีหน้าจัดการนักเรียน)

**การสร้างบัญชี**: admin สร้างเองผ่าน Supabase Dashboard (Authentication > Add user) ไม่มี
หน้าสมัครสมาชิกบนเว็บ (ตัดสินใจแล้ว — เหมาะกับกลุ่มครูเล็กในโรงเรียนเดียว ง่ายกว่ามาก)
บัญชีใหม่ทุกบัญชีได้ role='teacher' อัตโนมัติผ่าน trigger (`handle_new_user`) ต้องรัน SQL
`update profiles set role='admin' where email='...'` เองถึงจะเลื่อนเป็น admin

### ไฟล์ที่เกี่ยวข้อง
```
supabase-client.js   ตั้งค่า Supabase client กลาง + requireAuth()/signOut()/getProfile()
                      ทุกหน้า (entry/attendance/summary/manage/students/retention) import
                      จากไฟล์นี้แทนที่จะสร้าง client เองแยกกัน (เดิมมี URL/KEY hardcode ซ้ำ
                      4 ที่ ตอนนี้ที่เดียว)
login.html            หน้าล็อกอิน เรียก sb.auth.signInWithPassword() แล้วเด้งไป ?next= หรือ
                      entry.html — ถ้าล็อกอินอยู่แล้วข้ามหน้านี้ไปเลย (index.html = stub ที่
                      เด้งมาที่นี่เสมอ ดูหัวข้อ "Architecture ส่วนหน้าแรก/ล็อกอิน")
```
ทุกหน้าที่ต้องล็อกอินก่อนใช้ เรียก `const session = await requireAuth();` เป็นบรรทัดแรกของ
สคริปต์ — ถ้าไม่มี session จะเด้งไป `login.html?next=<หน้าปัจจุบัน>` อัตโนมัติ (ด่านหน้า
ฝั่ง frontend) ส่วนด่านหลังที่คุมจริงคือ RLS ที่ฐานข้อมูล (เปิดแล้ว — ดูด้านล่าง)

### ตาราง/ฟังก์ชันที่เพิ่ม (schema.sql เฟส 1)
```
profiles            id (=auth.users.id), email, name, role ('admin'/'teacher' default teacher)
subjects.owner_id   ครู/บัญชีที่เป็นเจ้าของวิชานั้น (uuid → auth.users) nullable
                     (วิชาเก่าก่อนมี auth ยังไม่มีเจ้าของ — admin แก้ได้อย่างเดียวจนกว่าจะ
                     ตั้งเจ้าของให้ผ่าน manage.html ช่อง "เจ้าของวิชา" เห็นเฉพาะ admin)

is_admin()                      -- true ถ้า role ของคนที่ล็อกอินอยู่ = admin
can_edit_subject(subject_id)    -- is_admin() หรือ owner_id ตรงกับคนล็อกอินอยู่
can_edit_unit/indicator/collection/session(...)  -- ไล่ join กลับไปเช็ค can_edit_subject
```
ทุกฟังก์ชัน `security definer` เพื่อไม่ให้ชนกับ RLS ของตารางที่มันอ่านเอง (pattern มาตรฐาน
ของ Supabase สำหรับ helper function ใน RLS policy)

### กติกา RLS (เปิดใช้งานจริงแล้ว — `enable row level security` รันแล้วทุกตาราง)
ทุกตาราง: **อ่านได้ถ้าล็อกอินแล้ว** (`auth.role() = 'authenticated'`) ไม่ว่าจะเป็นเจ้าของหรือ
ไม่ก็ตาม (ครูเห็นวิชาคนอื่นได้ ยืนยันแล้ว) — **แก้ไข/ลบ/เพิ่ม** ต้องผ่าน `can_edit_subject`
(หรือฟังก์ชันลูกที่ไล่กลับไปหา subject เดียวกัน) ยกเว้น `students` ที่ `is_admin()` เท่านั้น
ทดสอบแล้ว (2026-07-12): request ที่ไม่มี session (แค่ anon key) อ่าน `subjects` ได้ `[]`
และ PATCH ไม่มีผลอะไรเลย — ปิดช่องโหว่ "ใครมีลิงก์ก็แก้ข้อมูลได้" สำเร็จ

> **ทางฉุกเฉิน**: ถ้า RLS ทำเว็บพัง มีคำสั่ง `alter table X disable row level security;`
> ของทุกตาราง comment ไว้ท้ายสุดของ schema.sql พร้อมรันย้อนกลับได้ทันที

### ขั้นตอนเพิ่มบัญชีครูใหม่ (operational — ใช้ทุกครั้งที่มีครูคนใหม่เข้าระบบ)
```
1. Supabase Dashboard > Authentication > Add user — กรอกอีเมล+รหัสผ่านให้ครู (auto confirm)
   (trigger handle_new_user จะสร้างแถว profiles ให้อัตโนมัติ role='teacher' ทันที)
2. เข้า manage.html (ด้วยบัญชี admin) > เลือกวิชาที่ครูคนนั้นสอนทีละวิชา > ช่อง
   "เจ้าของวิชา" (เห็นเฉพาะ admin) > เลือกชื่อครูคนนั้น > บันทึก — ทำซ้ำทุกวิชาที่สอน
3. ครูคนนั้นล็อกอินที่ login.html ได้เลย เห็นทุกวิชา แต่แก้/ลบได้เฉพาะวิชาที่ตั้งเป็นเจ้าของ
   ในข้อ 2 (ถ้ายังไม่ตั้งเจ้าของ จะเห็นได้แต่แก้ไม่ได้ — ปุ่มแก้ไข/ลบจะซ่อนอัตโนมัติ)
```
เลื่อนใครเป็น admin เพิ่ม: `update profiles set role = 'admin' where email = '...';`

## Architecture ส่วนเกณฑ์เรียนซ้ำชั้น — implement แล้ว (`retention.html`)

เกณฑ์ 2 ข้อที่ผู้ใช้ยืนยัน (อิงระเบียบ ศธ. ว่าด้วยการเลื่อนชั้น/ซ้ำชั้น) — เข้าเกณฑ์ถ้าเข้า
**ข้อใดข้อหนึ่ง**:
```
ข้อ 1: เกรดเฉลี่ยทั้งปีการศึกษา (ถ่วงน้ำหนักด้วย credits, ไม่มี credits = ถือเป็น 1 หน่วย) < 1.00
ข้อ 2: จำนวนวิชาที่ติด 0 (เกรด 0 จากคะแนน) หรือ ร. หรือ มส. เกินครึ่งหนึ่งของรายวิชาที่
       ลงทะเบียนเรียนในปีการศึกษานั้น (นับเฉพาะวิชาบูรณาการ ไม่นับวิชาพื้นฐานสมาชิกซ้ำ)
```

> **ยืนยันกับผู้ใช้แล้ว (2026-07)**:
> - น้ำหนักเกรดเฉลี่ย: มี `credits` ใช้ `credits` ไม่มี (ส่วนใหญ่คือประถม) ถือเป็น 1 หน่วยเท่ากัน
> - ถ้านักเรียนคนไหนมีวิชาติด ร./มส. ที่ยังไม่แก้ไขอยู่แม้แต่วิชาเดียว **ไม่คิดเกรดเฉลี่ยทั้งปี
>   เลย** (ข้อ 1 = "ยังสรุปไม่ได้") แต่ข้อ 2 ยังเช็คได้ตามปกติ (ไม่ต้องรอข้อ 1)
> - วิชาบูรณาการกับวิชาพื้นฐานสมาชิกของมัน **นับเป็น 1 วิชา** (ตัวบูรณาการ) ไม่นับวิชาพื้นฐาน

> - รายงานนี้กรองด้วย `subjects.grade_level` + `subjects.year` (ปีการศึกษา) — ฟิลด์ `year`
>   **ไม่ได้บังคับกรอกตอนสร้างวิชา** (`manage.html` บังคับแค่ชื่อ+รหัสวิชา) ตอนนี้กรอกครบแค่
>   วิชาของ ม.3 เท่านั้น — ถ้าจะใช้รายงานนี้กับชั้นอื่น ต้องเข้า `manage.html` ไปเติม `year`
>   ให้ครบก่อน ไม่งั้น dropdown ปีการศึกษาของชั้นนั้นจะว่าง (หน้าเว็บมีข้อความเตือนไว้แล้ว)

### วิธีคำนวณ (`retention.html`, ใช้ฟังก์ชันจาก `supabase-client.js`)
```

   computeIntegratedResult() (วิชาบูรณาการ) — ฟังก์ชันเดียวกับที่ summary.html ใช้เป๊ะๆ
4. รายชื่อนักเรียนต่อวิชา = ผ่าน enrollments ของวิชาพื้นฐาน (วิชาบูรณาการดูผ่าน union ของ

5. รวมผลของนักเรียนแต่ละคนต่อทุกวิชาที่ลงทะเบียน แล้วเช็คเกณฑ์ 2 ข้อด้านบน
```

  เลือกองค์ประกอบมาตรฐาน → กำหนดคะแนนเต็ม → เพิ่มผู้เข้าร่วมทั้งห้องหรือทีละคน (คละชั้น/ห้องได้)
- `competency-entry.html` — เลือกรายการ → แต่ละองค์ประกอบมี “ครั้งที่ 1, 2, 3...” แยกของตัวเอง
  (ไม่ใช้วันที่ประเมิน) → เลือกครั้งที่แล้วกรอกคะแนนดิบรายคนผ่าน popup; รายชื่อในครั้งนั้นกรองตาม
  ช่วงชั้นขององค์ประกอบและ snapshot ไว้

ตารางใหม่ (`competency_assessments`, `competency_assessment_targets`, `competency_assessment_members`,
`competency_assessment_sessions`, `competency_assessment_session_students`,
`competency_assessment_scores`) แยกจากคะแนนรายวิชาทั้งหมด โดย session ผูก `target_id` และ
`attempt_no` (เลขครั้งต่อองค์ประกอบ) การสร้างครั้งประเมินจะ snapshot รายชื่อไว้ เพื่อไม่ให้ประวัติ
เปลี่ยนเมื่อครูแก้สมาชิกแม่แบบหรือนักเรียนเลื่อนชั้น

สิทธิ์: อ่านได้ทุกคนที่ล็อกอิน; เพิ่ม/แก้/ลบได้เฉพาะ admin หรือเจ้าของรายการนั้น (RLS ใหม่ใน
`schema.sql`) — ฐานข้อมูลที่เคยรัน schema รุ่นวันที่แล้วต้องรันเฉพาะบล็อก migration
“เปลี่ยนครั้งประเมินกิจกรรม/กิจวัตรจากตามวันที่” ที่ท้าย `schema.sql` ก่อน deploy หน้าใหม่

## งานถัดไป (ยังไม่ได้เริ่ม — เลือกได้ว่าจะทำอันไหนก่อน)

ไม่มีลำดับบังคับแล้ว (แผนเดิม + วิชาบูรณาการเสร็จหมดแล้ว) ตัวเลือกที่เหลือ เรียงตามที่คาดว่า
กระทบการใช้งานจริงของครูมากสุดก่อน:

1. วิชาเก่า `วิทยาการคำนวณ` ยังไม่มี `code`/`grade_level`/`owner_id` — ควรเข้า manage.html
   ไปเติมให้ครบ (owner_id ว่าง = แก้ได้แค่ admin คนเดียวตอนนี้)
2. **สรุปคะแนนสมรรถนะหลักจาก 3 แหล่งใน `summary.html`** — ตอนนี้ทำ schema และหน้ากำหนด/
   หน้ากรอกของกิจกรรมกับกิจวัตรแล้ว แต่ยังไม่เชื่อมผลขึ้น `summary.html`; ต้องตกลงรูปแบบการแสดง
   และสูตรรวมก่อน เพราะห้ามถัวเฉลี่ยแหล่งที่ยังไม่มีข้อมูลจนคะแนนต่ำลงผิดจริง
3. "ผ่อนผัน" แบบข้ามเช็ค มส. ทั้งหมด — ไม่ใช่แผนตอนนี้ (เลือกทำ "ชั่วโมงชดเชย" แทนแล้ว)
   เก็บไว้เป็นตัวเลือกสำรอง ยังไม่ confirm ว่าจะทำเมื่อไหร่

**Responsive shell + เมนูกลาง**: หน้าที่ต้องล็อกอิน 10 หน้าเรียก `app-shell.css` และ
`app-shell.js` ร่วมกัน — เมนูที่แสดงจริงกำหนดจาก `app-shell.js` เพียงจุดเดียว (อย่าเพิ่ม/แก้
รายการเมนูใน HTML รายหน้า), มือถือใช้แถบเมนูเลื่อนแนวนอนพร้อมปุ่มแตะสูงอย่างน้อย 44px, ฟอร์ม
2 คอลัมน์ยุบเป็นแนวตั้งที่จอ ≤600px และ iPad (≥700px) ใช้ความกว้างเนื้อหา 720px

## ข้อควรระวัง

- **Supabase free tier**: โปรเจกต์ auto-pause ถ้าไม่มีการใช้งานติดต่อกัน 7 วัน ต้องเข้า
  dashboard กด resume เอง ไม่มี auto-backup
- **RLS เปิดใช้งานจริงแล้ว** (ทั้งเฟส 1 และเฟส 2) ทดสอบแล้วว่าปิดช่องโหว่ anon เข้าถึงข้อมูล
  ได้จริง — ถ้าจะแก้ policy หรือ schema เพิ่มทีหลัง ระวังอย่าทำให้ policy เดิมหลุด/พังโดยไม่ตั้งใจ
  เตือนผู้ใช้ทุกครั้งก่อนแตะส่วนนี้ (ความเสี่ยง: ล็อกผู้ใช้จริงออกจากข้อมูลตัวเอง)
- **repo เป็น public** เพื่อใช้ GitHub Pages ฟรี — ใช้ได้แค่ Supabase publishable key
  (`sb_publishable_...`) ในโค้ดฝั่งนี้เท่านั้น ห้ามใส่ secret key เด็ดขาด
- **ข้อมูลวิชาเป็นของจริงแล้ว** (ไม่ใช่ seed อีกต่อไป) ผู้ใช้กรอกวิชาจริงไปแล้ว 25+ วิชา
  (ม.3 หลายเทอม มีทั้งวิชาพื้นฐานและบูรณาการ) — นักเรียน 4 คนที่เหลือยังเป็นข้อมูลตัวอย่างอยู่
  (รอหน้าจัดการนักเรียน) ระวังอย่าลบ/แก้ข้อมูลวิชาจริงโดยไม่ได้รับอนุญาตชัดเจน
- สูตร มส. ยืนยันแล้ว (อิงระเบียบ ศธ. + กติกาลา 2:1 ที่โรงเรียนกำหนดเพิ่ม) ใช้ทั้งประถมและมัธยม
  — "ผ่อนผัน" ยังไม่ implement

## หลักการทำงานร่วมกัน

- เป้าหมายตอนนี้คือทดสอบ workflow ให้ครูใช้จริงได้ก่อน ไม่ใช่สร้างครบทุกฟีเจอร์รอบเดียว
- คอมเมนต์ในโค้ดเป็นภาษาไทย ชื่อตาราง/ตัวแปรเป็นภาษาอังกฤษ ตามที่ schema.sql วางไว้
- แก้ไฟล์ใหญ่ ให้ส่งไฟล์เต็มแทนการทำ diff เล็กๆ (ลด regression risk)
- ล็อกการตัดสินใจแล้วอย่าเปิดถกใหม่ ถ้าจะเปลี่ยนต้องมีเหตุผลใหม่จริงๆ ไม่ใช่แค่ความชอบ
- ตัวเลข/เกณฑ์ที่กระทบเกรดจริงของนักเรียน (เกณฑ์เกรด, เกณฑ์ มส., นิยาม ร.) ห้ามเดาเอง
  ต้องถามผู้ใช้ยืนยันก่อนเสมอ

## บันทึกความคืบหน้าระหว่างรอบ

- อ่าน `WORKLOG.md` ทุกครั้งหลังอ่านไฟล์นี้ ก่อนเริ่มแก้ไข เพื่อรับบริบทที่เกิดขึ้นล่าสุด
- ก่อนแก้ไขงานใด ต้องอ่านโค้ดและส่วนที่เรียกใช้ซึ่งเกี่ยวข้องกับงานนั้นจริงเสมอ — เอกสารอธิบาย
  เจตนาและกติกา แต่โค้ดคือสภาพการทำงานปัจจุบัน ห้ามแก้จากเอกสารอย่างเดียว
- เมื่อแก้โค้ด รันการตรวจสอบ หรือมีการตัดสินใจสำคัญ ให้บันทึกวันที่ รายการเปลี่ยนแปลง
  และผลตรวจสอบไว้ต่อท้าย `WORKLOG.md` ก่อนส่งมอบงาน
- โฟลเดอร์นี้เป็นโฟลเดอร์พัฒนาเท่านั้น ห้ามอัปโหลดหรือแก้ไฟล์ในชุด deploy ที่ Google Drive
  เอง เว้นแต่ผู้ใช้สั่งชัดเจน
